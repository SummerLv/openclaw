# 02 - OpenClaw 整体架构

## 架构总览

OpenClaw 采用 **Gateway-centric + Plugin-driven** 架构：

│  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌────────┐ │
│  │ Channel  │  │  Agent   │  │  Plugin  │  │ Config │ │
│  │ Manager  │  │  Engine  │  │  Loader  │  │ Store  │ │
│  └────┬─────┘  └────┬─────┘  └────┬─────┘  └────────┘ │
│       │              │              │                    │
│  ┌────┴──────────────┴──────────────┴──────┐           │

│       │              │              │                    │
│  ┌────┴─────┐  ┌─────┴──────┐  ┌──┴──────────────┐   │
│  │ Provider │  │  Context   │  │   Memory        │   │

## 核心设计原则

1. **单一 Gateway**：每台主机一个 Gateway 进程，拥有所有消息面
2. **插件驱动**：核心保持精简，可选能力以插件形式交付
3. **SDK 边界**：`openclaw/plugin-sdk/*` 是唯一的跨包公共契约
4. **Manifest 优先**：插件的发现、验证、启用以元数据驱动

## 数据流

一条消息从进入到回复的完整流程：

```
用户消息 → Channel Plugin(入站)
         → Gateway(路由)
         → Agent Loop
             → Context Engine(组装上下文)
             → Model Provider(LLM 推理)
             → Tool Execution(工具执行)
             → Streaming Reply(流式回复)
         → Channel Plugin(出站)
         → 用户收到回复
```

## 核心子系统一览

| 子系统 | 位置 | 职责 |
|--------|------|------|
| **Gateway** | `src/gateway/` | 长驻守护进程，WS 服务器，所有消息面的所有者 |
| **Agent Engine** | `src/agents/` | Agent 运行时、Agent 循环、子 Agent |
| **Plugin System** | `src/plugins/` | 插件加载、注册、生命周期管理 |
| **Plugin SDK** | `src/plugin-sdk/` | 供插件开发者使用的公共 API |
| **Channels** | `src/channels/` + `extensions/` | 内置频道 + 插件频道 |
| **Context Engine** | `src/context-engine/` | 对话上下文构建与压缩 |
| **Memory** | `extensions/memory-*` | 记忆插件（独占槽位） |
| **Config** | `src/config/` | 用户配置、默认值、验证 |
| **Security** | `src/security/` | 认证、授权、沙箱 |
| **Secrets** | `src/secrets/` | API Key 等安全存储 |
| **CLI** | `src/cli/` + `src/commands/` | 命令行界面和斜杠命令 |
| **Sessions** | `src/sessions/` | 会话管理 |
| **Cron** | `src/cron/` | 定时任务调度 |
| **Media** | `src/media/` + `src/media-understanding/` | 媒体处理与理解 |

## 关键架构边界

### 插件与核心的边界
- 插件只能通过 `openclaw/plugin-sdk/*` 与核心交互
- 不得 import `src/**` 的任何文件
- 核心保持扩展无关，添加新插件不应要求修改核心代码

### 频道边界
- `src/channels/**` 是核心实现细节
- 插件作者需要新 seam 时，应加到 Plugin SDK

### Provider 边界
- 核心拥有通用推理循环
- Provider 插件通过注册和 hooks 拥有供应商特定行为
- 供应商特定的工具和设置属于所属插件
